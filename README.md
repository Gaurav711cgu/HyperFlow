**Designed for Senior ML / AI Engineer Portfolios | Targeted at Swiggy Instamart, Zomato, and Blinkit Roles**

Project Antigravity is a high-performance hyperlocal delivery ML engine built to solve **four documented operational gaps** publicly published by Swiggy Bytes and Zomato Engineering:
1. **Censored (Truncated) Demand**: Reconstructing real consumer demand when stockouts censor sales data.
2. **ETA Display Jitter**: Suppressing transient GPS anomalies without lagging behind real rider breakdown delays.
3. **Cancelled Order Resale Arbitrage**: Reselling cancelled food orders dynamically without allowing Sybil/co-location discount abuse.
4. **Cloud-Kitchen Proximity Collision**: Deflecting refund-loop fraud without triggering false positives for nearby buyers in multi-tenant cloud-kitchen hubs.

## 2. How We Claim & Verify the Results

Unlike standard portfolio mockups, the results claimed by Antigravity are derived from **first-principles mathematical models** executing locally:
*   **Tobit Parameter Recovery**: Evaluates how closely the Tobit MLE recovers the true latent demand coefficients ($\beta_{\text{true}}$) versus OLS. We calculate the **Wasserstein Distance** between the predicted demand distribution and the true demand distribution, proving that Tobit maintains distribution alignment while OLS fails.
*   **Residual Convergence**: ETA labels are calculated self-supervised post-hoc. A raw ETA bump is labeled "real" if the final delivery time converges to within 2 minutes of the raw prediction. This avoids using simulated labels that don't exist in live databases.
*   **OOD Storm Normalization**: Rather than testing on clean traffic, we drop the global velocity of the zone ($v_{\text{zone}}$) to simulate gridlock and storms. By normalizing rider speeds ($v_{\text{rider}} / v_{\text{zone}}$), we prove the Gated Smoother passes real storm-induced delays to the consumer while blocking GPS jumps.

## 3. Database Concurrency & System Design (Basic System Design)

To support 10k+ concurrent transactions at peak hours, Antigravity implements the following architecture:

*   **Row-Level Deadlock Prevention**: Orders reserving stock sorted by SKU ID lexicographically to avoid circular lock chains. We use `SELECT FOR UPDATE SKIP LOCKED` inside our PostgreSQL transactions to fail fast and fall back to adjacent dark stores rather than blocking connections.
*   **Atomic Redis Locking**: Acquires inventory holds using atomic `SET key value NX PX 500` calls, preventing lock leakage if FastAPI threads crash before releasing the key.
*   **Dual-Tier Feature Store**: Stores offline aggregates in PostgreSQL for training, while maintaining real-time features (like running zone average velocity) inside Redis for sub-10ms model inference.

## 4. Quick Start: Running Simulations & Platform Locally

### A. Run ML Simulations (CLI)
Verify the mathematical parameters and output reports in the `brain/` directory:
```bash
# 1. Test Tobit demand imputation under censoring rates (10%, 25%, 40%, 60%)
python3 -m ml_core.demand_simulation

# 2. Test ETA smoothing under storm surges
python3 -m ml_core.eta_simulation

# 3. Test dynamic cancelled order markdown and arbitrage guards
python3 -m ml_core.rescue_simulation

# 4. Test dispute triage and cloud-kitchen tenure bypass
python3 -m ml_core.fraud_simulation
```

### B. Launch Hugging Face Space Backend (FastAPI)
The backend loads all core modules and starts a REST API with CORS enabled:
```bash
pip install -r requirements.txt
python3 app.py
```
*App launches on `http://localhost:7860`.*

### C. Launch Vercel Frontend (React + Vite)
The frontend features a food slide background, interactive cart checkout, and 5 separate ML control room panels:
```bash
cd frontend
npm install
npm run dev
```
*App launches on `http://localhost:5173`.*

## 6. References & Engineering Literature Cited

To ensure Project Antigravity aligns with actual operational systems, the codebase is modeled directly on public engineering publications from Swiggy, Zomato, and Blinkit:

*   **Censored Demand (Instamart)**: 
    *   *Source*: *Swiggy Bytes (May 2023) - "Demand Forecasting at Instamart: Tackling Censored Sales."* 
    *   *Application*: Implemented the **Tobit Regression Type I log-likelihood solver** to impute latent demand on stockout days, resolving the downward bias caused by truncated sales records.
*   **ETA Calibration & Display Jitter**:
    *   *Source*: *Swiggy Bytes (May 2023) - "Display ETA Calibration: Smoothing Out Delivery Jumps."*
    *   *Application*: Swapped standard EWMA with a **Gated Random Forest Classifier** trained on post-hoc **Residual Convergence** to separate transient GPS noise from real delay signals.
*   **Cancelled Order Resale (CORO)**:
    *   *Source*: *Zomato Product Blog (Nov 2024) - "Introducing Food Rescue: Zero Waste Hyperlocal Operations."*
    *   *Application*: Modeled category-specific **Sensory Quality Index (SQI)** thermal decay curves and designed the **Anti-Arbitrage Sybil Check** (IP + proximity + cancel history) to close the discount cannibalization loops.
*   **Courier Batching & Dispatch**:
    *   *Source*: *Swiggy Tech (Dec 2022) - "Spatial Clustering & Constrained Batching for Courier Efficiency."*
    *   *Application*: Created the pre-calculated **SLA-pruned Nearest Neighbor Batcher** and **idle rider demand coordinates (hotspots)** to minimize fuel waste while maintaining the 15-minute delivery window.
